## 5. 工作留痕与阶段性汇报

### 5.1 工作留痕
- 所有你的工作成果（代码、配置、脚本等）必须保存在仓库中。
- 额外需要记录你的思考过程、遇到的技术难点及解决方案、每日进度等。这些记录统一存放在 **`pm/develop/`** 目录下。

### 5.2 阶段性汇报
每个阶段（例如每个任务排期中的里程碑）完成后，你必须在 `pm/develop/` 目录下创建一份汇报文档，命名格式为： YYYY-MM-DD_阶段名称_汇报.md

汇报内容必须包含：
- 本阶段完成的功能点（与需求文档对照）
- 本阶段编写的代码文件列表及行数统计
- 代码注释比例（确保≥50%）
- 遇到的主要问题与解决方案
- 下一步工作计划

## 6. 代码注释要求
- **注释比例必须占代码总行数的 50% 以上**。
- 注释包括：文件头说明（功能、作者、日期）、函数/类的功能、参数说明、返回值说明、复杂逻辑的行内解释、TODO 标记等。
- 不计算空行和纯括号行。可以使用工具（如 `cloc` 或自定义脚本）检查注释比例。
- 如果发现某个文件注释比例不足，需要补充注释直到达标。

## 7. 开发流程

### 7.1 每日工作循环
1. **拉取最新代码**：`git pull`
2. **阅读 `pm/` 下的需求与排期**：确认当前优先级最高的任务。
3. **设计/编码**：按照需求编写代码，边写边加注释。
4. **本地测试**：确保功能正常，没有破坏已有功能。
5. **提交代码**：调用 skill `/commit` 提交并推送代码
6. **编写工作记录**：在 `pm/develop/` 下更新或创建当天的进度记录。

### 7.2 阶段完成时
- 除了提交代码，还需要创建“阶段性汇报”文档（见 5.2）。
- 主动在汇报文档中提醒项目经理，说明本阶段已就绪，请求下一步指示或需求确认。

## 8. 注意事项
- 不要向任何人泄露 API Key，避免将 Key 写入代码或提交到仓库。
- 如果遇到需求模糊或不合理，在汇报中提出疑问，等待项目经理澄清后再继续开发。
- 保持代码风格一致，推荐使用 ESLint / Prettier（根据项目技术栈）。
- 每次 push 前确保代码不会破坏现有功能。
